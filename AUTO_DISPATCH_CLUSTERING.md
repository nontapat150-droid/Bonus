# 📍 ระบบแจกจ่ายงานแบบ Clustering - เอกสารอัปเดต

## 🎯 ปัญหาที่แก้ไข

**เดิม**: เมื่อกดแจกจ่ายงานอัตโนมัติ ระบบใช้ **Greedy Nearest** คือเลือกงานที่ใกล้ที่สุดไปเรื่อย ๆ
- ⚠️ **ผลลัพธ์**: บริเวณเดียวกันมีหลายทีมปะปนกัน
- ❌ ทีม A อยู่ในบริเวณเหนือ แต่ได้งานทั้งเหนือ ใต้ ตะวันออก

**ใหม่**: เมื่อกดแจกจ่ายงานอัตโนมัติ ระบบใช้ **Geographical Clustering**
- ✅ จัดกลุ่มงานตามพิกัด (ใกล้กัน = กลุ่มเดียว)
- ✅ แต่ละกลุ่มแจกให้ทีมเดียว
- ✅ **ผลลัพธ์**: บริเวณของทีม A มีเฉพาะทีม A, ทีม B มีเฉพาะทีม B

## 🔧 อัลกอริทึมใหม่: Strict Quota Clustering

### ขั้นตอนที่ 1: เลือก Seed Points
```
เลือก seed point แต่ละ cluster ตามลำดับ:
1. หากทีมมีงานอยู่แล้ว (Anchor) ใช้จุดศูนย์กลางของงานเดิมเป็น Seed
2. หากยังไม่มี ใช้จุดที่ห่างจากกลุ่มอื่นมากที่สุด (Farthest Point)
```

### ขั้นตอนที่ 2: Assign Jobs to Clusters แบบระบุโควตา (Strict Quota)
```
วนลูปตามจำนวนงานที่เหลือ:
  - หาคู่ (งาน, ทีม) ที่มีระยะทางใกล้ที่สุดจากทุกคู่ที่เป็นไปได้
  - ตรวจสอบโควตาของทีม (Limit) ถ้าทีมไหนงานเต็มแล้ว ให้ข้ามไป
  - แจกงานให้ทีมที่ใกล้ที่สุด
  - Update centroid ของ cluster ใหม่โดยถ่วงน้ำหนักกับ Anchor เดิม

ทำซ้ำจนกว่าทุกทีมจะได้งานครบตามโควตา
```

### ขั้นตอนที่ 3: Assign Clusters to Teams
```
Cluster 1 → ทีม A
Cluster 2 → ทีม B
Cluster 3 → ทีม C
...
```

## 📊 ตัวอย่างผลลัพธ์

### ก่อนแจกจ่าย (Scattered)
```
ทีม A: AC-001(13.7°N), AC-002(13.4°N), AC-003(13.6°N)
ทีม B: BC-001(13.6°N), BC-002(13.5°N)

❌ งานของทีม A กระจัดกระจาย
❌ งานของทีม B ยุ่งเกี่ยวกับ A
```

### หลังแจกจ่าย (Clustered)
```
ทีม A: AC-001(13.76°N), AC-002(13.75°N), AC-003(13.77°N)
ทีม B: BC-001(13.45°N), BC-002(13.46°N)

✅ ทีม A อยู่บริเวณเหนือ (13.75-13.77°N)
✅ ทีม B อยู่บริเวณใต้ (13.45-13.46°N)
✅ ไม่มีการทับซ้อน
```

## 💡 ข้อดีของ Clustering

| ข้อดี | รายละเอียด |
|-------|-----------|
| **ประสิทธิภาพ** | งานที่ใกล้กัน = ประหยัดเวลาและน้ำมัน |
| **ความชัดเจน** | แต่ละทีมมีบริเวณเดียวกัน ไม่สับสน |
| **การบริหาร** | แอดมินรู้ว่าทีมไหนอยู่บริเวณไหน |
| **ความพึงพอใจ** | งานติดตั้งเป็นกลุ่มใจเย็น ไม่วิ่งกลับมา |

## 🧪 ทดสอบ

1. **กรณี 1: 2 ทีม, 10 งาน**
   ```
   ✅ ควรได้ 2 clusters ที่แยกกัน
   ✅ ทีม A ได้ 5 งาน, ทีม B ได้ 5 งาน
   ✅ งานของ A อยู่บริเวณเดียวกัน
   ```

2. **กรณี 2: 3 ทีม, 6 งาน (1 ทีมไม่มีงาน)**
   ```
   ✅ ได้ 2 clusters (เพราะเฉพาะ 2 ทีมที่มีโควตา)
   ✅ 1 ทีมไม่ได้งาน
   ```

3. **กรณี 3: เมื่อ limit > total unassigned jobs**
   ```
   ✅ ทีมที่มี limit สูงสุดได้งานพอ
   ✅ ทีมอื่นไม่ได้งาน
   ```

## 🚀 Performance

- **Speed**: O(n²) สำหรับ clustering (n = จำนวนงาน)
- **Memory**: O(n) สำหรับเก็บ clusters
- **Typical case**: 100 jobs, 3 teams = ~1 วินาที

---

## 🐛 Known Limitations

1. **Clusters ไม่เท่ากัน**: ถ้างานกระจายไม่สม่ำเสมอ
   - ปก: ใช้ greedy assignment พูดคุยกับหน่วยสนับสนุน

2. **Edge case - งานไม่พอ**:
   - ถ้า unassigned jobs < จำนวนทีม → clusters บางอันอาจว่าง
   - ปก: ระบบจัดการได้ ได้งาน 0 jobs

3. **ไม่คำนึงถึงเวลา**:
   - ระบบแจกจ่ายตามพิกัดเท่านั้น ไม่คำนึงถึงความยากของงาน
   - ปก: อาจจะเพิ่มในอนาคต

## 📞 FAQ

**Q: ทำไมงานของทีมเดิมยังปนกันอยู่?**
- A: Clustering ใช้เฉพาะสำหรับ **unassigned jobs** ที่จ่ายใหม่ งานเก่าของทีมยังอยู่ที่เก่า

**Q: สามารถเปลี่ยน cluster size ได้หรือ?**
- A: ปัจจุบันยังไม่มี UI เพื่อปรับ ต้องแก้ไขโค้ด หรือปรึกษาแอดมิน

**Q: การสร้าง route ใหม่มีผลต่อ map_link ไหม?**
- A: ใช่ map_link ถูกสร้างใหม่ตามลำดับที่ buildNearestRoute จัดเรียง
